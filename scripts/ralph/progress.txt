## Codebase Patterns
- Put schema changes in `db/migrations` and verify them with dependency-free `node:sqlite` checks under `scripts/` and `tests/`.
- Make migrations rerunnable with `IF NOT EXISTS`; `scripts/verify-workspace-schema.cjs` applies the same migration twice to catch duplicate-DDL failures.
- When SQLite needs rerunnable schema expansion beyond `CREATE TABLE` or `CREATE INDEX`, prefer additive side tables over `ALTER TABLE ... ADD COLUMN` because the bundled engine here does not support `ADD COLUMN IF NOT EXISTS`.
- Apply schema setup through `src/db/migrations.cjs` so tests and verifiers always run the full ordered migration set instead of hard-coding a single SQL file.
- `scripts/lint.cjs` recursively syntax-checks `.cjs` files under `scripts`, `src`, and `tests`, so new modules are automatically covered by `npm run check`.
- Model SQLite booleans as constrained integers like `is_active INTEGER CHECK (is_active IN (0, 1))`, and export the shared defaults from the matching `src/db/schema/*.cjs` module so tests verify the migration contract directly.
- For tenant-scoped child tables, add a unique parent index on `(workspace_id, id)` before introducing composite foreign keys that must block cross-workspace references.
- Use partial unique indexes for nullable dedupe keys such as `source_url`, `canonical_url`, and hashes so SQLite enforces idempotency without treating multiple `NULL` values as conflicts.
- Use partial unique indexes on nullable scope foreign keys to model one workspace default row plus one target override row, such as `alert_policy` records with `monitoring_target_id IS NULL` versus `IS NOT NULL`.
- For backend write flows, wrap multi-table SQLite mutations in an explicit transaction helper and inject `now`/ID/token generators so `node:test` coverage stays deterministic without mocking the database.
- For workspace-owned backend writes, gate creation on an `active` `workspace_membership` row and persist seeded child records in the same transaction, using request order for fields like `display_order`.
- For background jobs that call external services, inject the search/LLM adapters plus `now`/ID generators, and keep the final SQLite persistence step in a short transaction so reruns stay deterministic and easy to test.
- For canonical article ingestion, keep dedupe identity in `article` and fetched body/source fields in a 1:1 `article_content` table so rerunnable migrations can add content storage without rebuilding the base article table.
- For ordered keyword collections, treat `display_order` as scoped to a single `source_type`, and make reorder flows rewrite the full ID set for that source in one transaction.
- When a target discovery rerun regenerates profile data, clear any saved review state in the same transaction so stale mismatch or approval decisions do not leak into the next review cycle.
- For rerunnable analysis stages, keep one stable `article_analysis` row per `(workspace_id, monitoring_target_id, article_id)` and store fan-out evidence in a stage-specific side table so later enrichments can update their own fields without clobbering other analysis outputs.
- For analysis stages that should not rerun on every scheduler pass, left-join `article_analysis` in the eligibility query and accept explicit `{ workspaceId, monitoringTargetId, articleId }` reanalysis keys so completed stage output is only replaced on demand.
- When an analysis stage emits an ordered label list, persist the ordered JSON snapshot on `article_analysis` and replace the normalized row-per-label side table in the same transaction so reruns stay query-friendly and idempotent.
- Keep derived score-band mapping in shared schema helpers and reuse those exports in backend jobs so runtime classification stays aligned with SQLite `CHECK` constraints.
- Resolve alert settings from `alert_policy` with target scope first, then workspace scope, and finally the monitoring target's default threshold with all channels disabled when no saved policy exists.
- For auditable alert dispatch jobs, insert the parent `alert_event` first, call external channel adapters outside the transaction, then persist per-channel `alert_delivery` records plus 1:1 dispatch metadata and the final event status in a second transaction.
- For aggregate notification jobs derived from existing `alert_event` rows, keep batching state in additive side tables keyed back to source event IDs, and enforce a unique `(workspace_id, alert_event_id)` membership so reruns cannot rebatch the same immediate alert.
- For article read APIs, treat `article_analysis` as the feed grain, use `article_analysis_topic_label` for topic filtering, and deserialize the ordered `article_analysis.topic_labels` JSON snapshot for the response payload.
- Use `article_analysis.id` as the stable key for article detail and drilldown queries, because one canonical `article` can legitimately appear under multiple monitoring targets in the same workspace.
- For analytics aggregations, query high-risk `article_analysis` rows with non-null `article_content.published_at`, normalize publisher/reporter labels case-insensitively, and return `articleAnalysisIds` with `articleIds` so drilldowns stay aligned with the feed grain.
- For lightweight UI stories in this repo, keep SQL and auth checks in the existing service module, render HTML in a small `src/ui/*.cjs` view, and wrap both with a dependency-free `node:http` route so `node:test` and browser demos exercise the same surface.

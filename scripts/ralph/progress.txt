## Codebase Patterns
- Put schema changes in `db/migrations` and verify them with dependency-free `node:sqlite` checks under `scripts/` and `tests/`.
- Make migrations rerunnable with `IF NOT EXISTS`; `scripts/verify-workspace-schema.cjs` applies the same migration twice to catch duplicate-DDL failures.
- Apply schema setup through `src/db/migrations.cjs` so tests and verifiers always run the full ordered migration set instead of hard-coding a single SQL file.
- `scripts/lint.cjs` recursively syntax-checks `.cjs` files under `scripts`, `src`, and `tests`, so new modules are automatically covered by `npm run check`.
- Model SQLite booleans as constrained integers like `is_active INTEGER CHECK (is_active IN (0, 1))`, and export the shared defaults from the matching `src/db/schema/*.cjs` module so tests verify the migration contract directly.
- For tenant-scoped child tables, add a unique parent index on `(workspace_id, id)` before introducing composite foreign keys that must block cross-workspace references.
- Use partial unique indexes for nullable dedupe keys such as `source_url`, `canonical_url`, and hashes so SQLite enforces idempotency without treating multiple `NULL` values as conflicts.
- Use partial unique indexes on nullable scope foreign keys to model one workspace default row plus one target override row, such as `alert_policy` records with `monitoring_target_id IS NULL` versus `IS NOT NULL`.
- For backend write flows, wrap multi-table SQLite mutations in an explicit transaction helper and inject `now`/ID/token generators so `node:test` coverage stays deterministic without mocking the database.
- For workspace-owned backend writes, gate creation on an `active` `workspace_membership` row and persist seeded child records in the same transaction, using request order for fields like `display_order`.
- For background jobs that call external services, inject the search/LLM adapters plus `now`/ID generators, and keep the final SQLite persistence step in a short transaction so reruns stay deterministic and easy to test.
- When a target discovery rerun regenerates profile data, clear any saved review state in the same transaction so stale mismatch or approval decisions do not leak into the next review cycle.

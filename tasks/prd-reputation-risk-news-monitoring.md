# PRD: 평판 리스크 뉴스 모니터링 플랫폼

## 1. Introduction / Overview

대기업 홍보실, 연예 기획사, 인플루언서 매니저 조직은 특정 기업 또는 인물에 관한 온라인 뉴스 기사를 상시 모니터링해야 한다. 특히 기업 오너 일가, 경영진, 연예인, 인플루언서와 관련된 기사 중에서 명예훼손 우려, 허위사실 유포 가능성, 민감한 정보 확산, 도덕성 비판 등 평판 리스크가 높은 기사를 빠르게 탐지하고 대응하는 것이 중요하다.

현재는 사람이 포털과 검색 엔진을 반복적으로 확인하며 기사를 읽고 분류해야 하므로 피로도가 높고, 기사 발행 속도를 따라가기 어렵다. 본 기능은 여러 고객사가 사용하는 멀티테넌트 SaaS 형태로 제공되며, 사용자가 모니터링 대상과 키워드를 등록하면 시스템이 네이버, 네이트, 구글 뉴스 검색 결과와 원문 기사를 수집하고, 기사 중복 제거, 주제 분류, 요약, 위험도 점수화(0~100), 판정 근거 생성, 즉시 알림을 자동으로 수행한다.

v1의 핵심은 "사람이 전부 읽기 전에 고위험 기사를 먼저 보여주고 즉시 알리는 것"이다. 최종 법적 판단은 사용자가 내리며, 시스템은 우선순위화와 빠른 탐지를 지원한다.

## 2. Goals

- 사용자가 모니터링 대상(기업, 인물)을 등록하고 운영 가능한 상태로 확정할 수 있어야 한다.
- 등록된 대상에 대해 포털 검색 결과와 원문 기사를 자동 수집하고 중복 없이 저장해야 한다.
- 각 기사에 대해 대상 일치 여부, 주제 카테고리, 요약, 위험도 점수, 판정 근거를 생성해야 한다.
- 위험도 임계값 이상 기사에 대해 슬랙, 이메일, SMS로 즉시 알림을 보낼 수 있어야 한다.
- 사용자가 실시간 기사 현황과 카테고리/언론사/기자 집계를 대시보드에서 확인할 수 있어야 한다.
- 사람이 수동으로 수행하던 반복 모니터링 시간을 의미 있게 줄여야 한다.

## 3. User Stories

### US-001: 워크스페이스 생성 및 팀원 초대
**Description:** As a 홍보실 관리자, I want my organization to have its own isolated workspace so that our monitoring data is separated from other customers.

**Acceptance Criteria:**
- [ ] 신규 고객사는 고유한 워크스페이스를 생성할 수 있다.
- [ ] 워크스페이스 간 기사, 키워드, 알림 설정, 분석 결과가 서로 조회되지 않는다.
- [ ] 관리자 사용자는 이메일로 팀원을 초대할 수 있다.
- [ ] 초대된 사용자는 자신의 워크스페이스 데이터만 조회할 수 있다.
- [ ] Typecheck/lint passes

### US-002: 모니터링 대상 등록
**Description:** As a 사용자, I want to register a company or person as a monitoring target so that the system knows what to track.

**Acceptance Criteria:**
- [ ] 사용자는 모니터링 대상 유형을 `기업` 또는 `인물`로 선택할 수 있다.
- [ ] 사용자는 기본 키워드를 하나 이상 입력해야 저장할 수 있다.
- [ ] 대상별 표시 이름, 설명 메모, 기본 알림 임계값을 설정할 수 있다.
- [ ] 저장 후 대상 상태는 `검토 필요`로 생성된다.
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-003: 대상 분석 결과 검토 및 확정
**Description:** As a 사용자, I want to review the system's understanding of a target before activation so that unrelated 기사 수집을 줄일 수 있다.

**Acceptance Criteria:**
- [ ] 시스템은 웹 검색과 LLM을 사용해 대상 후보 설명과 관련 엔티티를 생성한다.
- [ ] 화면에는 대상 요약, 연관 인물/기업, 별칭, 대표 키워드 후보가 표시된다.
- [ ] 사용자는 `일치함`, `부분 일치`, `불일치` 중 하나로 검토 결과를 기록할 수 있다.
- [ ] 사용자가 `불일치`를 선택하면 수동 키워드 편집 단계로 이동할 수 있다.
- [ ] 대상이 확정되기 전까지 자동 수집은 시작되지 않는다.
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-004: 키워드 확장 및 수동 편집
**Description:** As a 사용자, I want the system to expand my seed keywords and let me edit them so that search coverage improves without losing control.

**Acceptance Criteria:**
- [ ] 시스템은 사용자 입력 키워드를 기반으로 확장 키워드 후보를 생성한다.
- [ ] 확장 키워드는 사용자 원본 입력과 구분되어 저장된다.
- [ ] 사용자는 확장 키워드를 추가, 삭제, 비활성화할 수 있다.
- [ ] 사용자는 제외 키워드(negative keyword)를 입력해 불필요한 수집을 줄일 수 있다.
- [ ] 대상이 `활성` 상태가 되면 확정된 키워드 집합이 수집 작업에 사용된다.
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-005: 포털 기반 기사 후보 수집
**Description:** As a 운영 시스템, I want to search portal news results for active targets so that newly published articles can be discovered quickly.

**Acceptance Criteria:**
- [ ] 활성화된 대상마다 네이버, 네이트, 구글 뉴스 검색을 주기적으로 수행한다.
- [ ] 검색 시 대상별 활성 키워드와 제외 키워드를 함께 반영한다.
- [ ] 각 검색 결과에 대해 제목, 요약문, 노출 시각, 포털 링크, 원문 링크를 저장한다.
- [ ] 동일 검색 배치가 재실행되어도 중복 후보가 무한히 누적되지 않는다.
- [ ] 수집 실패 건은 재시도 가능 상태로 기록된다.
- [ ] Typecheck/lint passes

### US-006: 원문 수집 및 멱등 저장
**Description:** As a 운영 시스템, I want to fetch original article pages and store them idempotently so that analysis uses full article text without duplicate records.

**Acceptance Criteria:**
- [ ] 포털 검색 결과에서 원문 링크가 있으면 원문 페이지를 추가 수집한다.
- [ ] 기사 본문, 발행 시각, 기자명, 언론사명, 조회수 등 추출 가능한 메타데이터를 저장한다.
- [ ] 동일 기사 URL 또는 동일 본문 해시의 기사는 하나의 canonical article로 병합한다.
- [ ] 재수집 시 기존 기사 레코드를 갱신하되 중복 article ID를 생성하지 않는다.
- [ ] 원문 수집 실패 시 실패 사유가 기록된다.
- [ ] Typecheck/lint passes

### US-007: 기사 분석 파이프라인 실행
**Description:** As a 사용자, I want each collected article to be analyzed automatically so that I can quickly understand risk and context.

**Acceptance Criteria:**
- [ ] 각 기사는 대상 일치 여부 또는 대상 연관성 점수를 생성한다.
- [ ] 각 기사는 1개 이상의 주제 카테고리로 분류된다.
- [ ] 각 기사는 3문장 이내 요약을 생성한다.
- [ ] 각 기사는 0~100 위험도 점수와 위험도 구간(`낮음`, `중간`, `높음`)을 가진다.
- [ ] 위험도 판정 근거는 사람이 읽을 수 있는 짧은 설명으로 저장된다.
- [ ] 분석 결과에는 사용된 모델/버전 정보가 함께 기록된다.
- [ ] Typecheck/lint passes

### US-008: 알림 정책 및 채널 설정
**Description:** As a 사용자, I want to configure thresholds and channels per workspace or target so that alerts match my team's operating process.

**Acceptance Criteria:**
- [ ] 사용자는 워크스페이스 기본 위험도 임계값을 설정할 수 있다.
- [ ] 사용자는 대상별로 임계값을 별도 지정할 수 있다.
- [ ] 사용자는 슬랙, 이메일, SMS 채널을 각각 활성화 또는 비활성화할 수 있다.
- [ ] 사용자는 채널별 수신 대상(웹훅, 이메일 주소, 휴대폰 번호)을 저장할 수 있다.
- [ ] 유효하지 않은 채널 설정은 저장 전에 검증된다.
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-009: 고위험 기사 즉시 알림
**Description:** As a 사용자, I want to be notified immediately when a high-risk article is detected so that I can respond before the issue spreads.

**Acceptance Criteria:**
- [ ] 기사 위험도 점수가 활성 임계값 이상이면 알림 작업이 자동 생성된다.
- [ ] 알림에는 대상명, 기사 제목, 위험도 점수, 위험도 근거, 언론사, 링크가 포함된다.
- [ ] 1분 이내 다수의 고위험 기사가 발생하면 묶음 알림을 생성할 수 있다.
- [ ] 묶음 알림에는 최고 위험 기사와 요약 집계 정보가 포함된다.
- [ ] 각 채널별 발송 성공/실패 상태가 기록된다.
- [ ] Typecheck/lint passes

### US-010: 실시간 모니터링 대시보드 조회
**Description:** As a 사용자, I want to see newly collected and analyzed articles in one dashboard so that I can triage the queue quickly.

**Acceptance Criteria:**
- [ ] 대시보드 목록에는 기사 제목, 대상명, 위험도, 카테고리, 언론사, 기자, 발행 시각이 표시된다.
- [ ] 사용자는 대상, 위험도 구간, 카테고리, 기간, 언론사 기준으로 필터링할 수 있다.
- [ ] 사용자는 위험도 높은 순 또는 최신순으로 정렬할 수 있다.
- [ ] 기사 상세 화면에서 요약, 판정 근거, 매칭 키워드, 원문 링크를 볼 수 있다.
- [ ] 새 기사 유입 시 화면 새로고침 없이 목록을 갱신할 수 있다.
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

### US-011: 카테고리 및 발행 주체 분석
**Description:** As a 사용자, I want analytics by category, publisher, and reporter so that I can identify recurring risk patterns.

**Acceptance Criteria:**
- [ ] 기간별 고위험 기사 수를 카테고리 기준으로 집계해 표시한다.
- [ ] 고위험 기사를 가장 많이 발행한 언론사 순위를 표시한다.
- [ ] 고위험 기사를 가장 많이 작성한 기자 순위를 표시한다.
- [ ] 특정 대상 기준으로 집계 결과를 좁혀서 볼 수 있다.
- [ ] 집계 화면에서 원본 기사 목록으로 드릴다운할 수 있다.
- [ ] Typecheck/lint passes
- [ ] Verify in browser using dev-browser skill

## 4. Functional Requirements

1. **FR-1:** 시스템은 여러 고객사가 동시에 사용할 수 있는 멀티테넌트 워크스페이스 구조를 제공해야 한다.
2. **FR-2:** 시스템은 워크스페이스 단위로 사용자, 모니터링 대상, 기사, 알림 설정, 분석 결과를 분리 저장해야 한다.
3. **FR-3:** 사용자는 모니터링 대상을 `기업` 또는 `인물` 타입으로 등록할 수 있어야 한다.
4. **FR-4:** 모니터링 대상 등록 시 시스템은 웹 검색과 LLM을 이용해 대상 설명, 연관 엔티티, 별칭, 확장 키워드 후보를 생성해야 한다.
5. **FR-5:** 사용자는 자동 생성된 확장 키워드를 검토, 수정, 삭제, 비활성화할 수 있어야 한다.
6. **FR-6:** 시스템은 사용자 원본 키워드와 LLM이 확장한 키워드를 구분하여 저장해야 한다.
7. **FR-7:** 시스템은 제외 키워드를 지원해 특정 표현이 포함된 기사 수집을 줄일 수 있어야 한다.
8. **FR-8:** 대상이 사용자의 검토를 통해 확정되기 전까지 자동 기사 수집을 시작하면 안 된다.
9. **FR-9:** 시스템은 활성 대상에 대해 네이버, 네이트, 구글 뉴스 검색 결과를 주기적으로 조회해야 한다.
10. **FR-10:** 시스템은 포털 검색 결과에서 원문 기사 링크를 추적해 원문 페이지를 수집해야 한다.
11. **FR-11:** 시스템은 기사 제목, 본문, 발행 시각, 기자, 언론사, 링크, 조회수 등 추출 가능한 메타데이터를 저장해야 한다.
12. **FR-12:** 시스템은 동일 URL, canonical URL, 또는 본문 해시를 이용해 기사 저장의 멱등성을 보장해야 한다.
13. **FR-13:** 시스템은 각 기사에 대해 대상 일치 여부 또는 연관성 점수를 계산해야 한다.
14. **FR-14:** 시스템은 각 기사에 대해 주제 카테고리를 하나 이상 부여해야 한다.
15. **FR-15:** 시스템은 각 기사에 대해 0~100 위험도 점수, 위험도 구간, 판정 근거를 생성해야 한다.
16. **FR-16:** 시스템은 분석 결과와 함께 사용된 모델 버전 및 분석 시각을 기록해야 한다.
17. **FR-17:** 사용자는 워크스페이스 기본 임계값과 대상별 임계값을 설정할 수 있어야 한다.
18. **FR-18:** 시스템은 슬랙, 이메일, SMS를 알림 채널로 지원해야 한다.
19. **FR-19:** 시스템은 위험도 점수가 임계값 이상인 기사를 탐지하면 즉시 알림을 생성해야 한다.
20. **FR-20:** 시스템은 짧은 시간 안에 여러 고위험 기사가 발생한 경우 묶음 알림을 생성할 수 있어야 한다.
21. **FR-21:** 시스템은 각 알림의 채널별 발송 결과와 실패 사유를 기록해야 한다.
22. **FR-22:** 시스템은 실시간 대시보드에서 기사 목록, 필터, 정렬, 상세 보기 기능을 제공해야 한다.
23. **FR-23:** 시스템은 카테고리, 언론사, 기자 기준의 집계 분석 화면을 제공해야 한다.
24. **FR-24:** 사용자는 집계 결과에서 원본 기사 목록으로 드릴다운할 수 있어야 한다.
25. **FR-25:** 시스템은 사용자가 기사 수집 또는 분석 실패 건을 재처리할 수 있는 운영 상태 정보를 보존해야 한다.

## 5. Non-Goals (Out of Scope)

- 커뮤니티, SNS, 유튜브, 블로그, 댓글 영역까지 모니터링 범위를 확장하는 기능
- 기사 삭제 요청, 정정 요청, 소송 문안 작성 등 법무 대응 자동화
- 사람의 개입 없이 엔티티 일치 여부를 완전 자동으로 확정하는 기능
- PDF 리포트 생성, 보고서 공유 링크, 외부 고객용 리포트 포털
- 해외 언어권 매체 전반을 대상으로 한 다국어 글로벌 모니터링
- TV 방송, 라디오, 팟캐스트 전사 데이터 수집

## 6. Design Considerations

- 신규 대상 등록은 `기본 정보 입력 -> 대상 분석 결과 검토 -> 키워드 확정 -> 활성화`의 단계형 흐름이어야 한다.
- 실무 사용자는 "새 기사 확인"보다 "고위험 기사 우선 확인"이 중요하므로 기본 정렬은 고위험 우선이 적합하다.
- 기사 상세 화면에는 원문 링크, 요약, 판정 근거, 관련 키워드가 한 화면에 보여야 한다.
- 집계 화면은 차트만 보여주는 방식보다, 클릭 시 기사 목록으로 즉시 이동할 수 있는 운영형 UI가 적합하다.
- 알림 메시지는 모바일 환경에서도 빠르게 읽을 수 있도록 5줄 안팎의 핵심 정보 중심으로 구성해야 한다.

## 7. Technical Considerations

- 포털 수집, 원문 수집, 분석, 알림 발송은 서로 분리된 비동기 파이프라인으로 구성하는 것이 바람직하다.
- 기사 중복 제거는 단순 URL 비교만으로 충분하지 않으므로 canonical URL, 정규화 제목, 본문 해시를 함께 고려해야 한다.
- 위험도 평가는 LLM 기반으로 수행하되, 동일 기사 재분석 비용을 줄이기 위한 캐싱 정책이 필요하다.
- 주제 분류는 재현 가능한 기준이 있어야 하므로, 업계에서 널리 사용하는 토픽 모델링 기법 또는 검증 가능한 분류 체계를 선정해야 한다.
- 멀티테넌트 SaaS이므로 데이터 파티셔닝, 고객별 설정 분리, 감사 로그가 중요하다.
- 원문 기사 HTML 보관 여부와 보관 기간은 법무 및 저장 비용 관점에서 별도 정책이 필요하다.
- SMS는 채널 비용이 높으므로 기본적으로 고위험 기사 또는 묶음 알림에만 제한하는 운영 정책을 검토해야 한다.
- 검색 결과와 원문 HTML 구조가 자주 바뀔 수 있으므로 수집기 모듈별 장애 감지와 빠른 수정 체계가 필요하다.

## 8. Success Metrics

- 활성 대상에 대해 새 기사 후보가 검색 결과에 나타난 뒤 95% 이상이 2분 이내 시스템에 저장된다.
- 저장된 기사 중 95% 이상이 수집 후 5분 이내에 분석 완료 상태가 된다.
- 임계값 이상 기사에 대한 알림은 분석 완료 후 95% 이상이 1분 이내 발송된다.
- 수동 라벨링한 검증 셋에서 `고위험` 판정(예: 80점 이상)의 precision이 85% 이상이다.
- 파일럿 고객 기준으로 기존 수동 모니터링 대비 기사 검토 시간이 50% 이상 감소한다.
- 사용자는 특정 대상의 최근 고위험 기사 상위 10건을 30초 이내에 찾을 수 있다.

## 9. Open Questions

- v1의 기본 언어 범위를 한국어 기사로 한정할지, 영어 기사 일부까지 포함할지 결정이 필요하다.
- 고위험 기사 카테고리 taxonomy를 고정할지, 고객별 커스텀 카테고리를 허용할지 결정이 필요하다.
- 토픽 모델링 기준 모델을 BERTopic 계열, 분류 모델 기반 taxonomy, 또는 하이브리드 방식 중 무엇으로 둘지 결정이 필요하다.
- SMS 알림은 모든 고위험 기사에 보낼지, 최고 위험 구간에만 보낼지 운영 정책 결정이 필요하다.
- 기사 조회수, 댓글 수 등 메타데이터는 매체별 편차가 큰데, 누락 허용 규칙을 어디까지 둘지 결정이 필요하다.
- 법무/홍보팀이 기사 상태를 `검토 중`, `대응 완료`, `무시`로 표시하는 후속 워크플로를 v1.1에 포함할지 검토가 필요하다.
